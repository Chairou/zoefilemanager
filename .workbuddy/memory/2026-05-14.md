m_tabs[idx].fileList->setActiveHighlight()` 访问已销毁的 FileListView，崩在 palette()。
**修复顺序**：先 m_tabs.removeAt → setCurrentIndex 到邻居 → removeTab → delete w。

### SFTP 复制 / 删除 / 改名（新增）
- `SftpClient` 增加：`removeFile`、`removeEmptyDirectory`、`removeRecursive`、
  `renamePath`、`makeDirectory`、`copyFile`（按 32K 块流式 read+write）、`copyRecursive`
- `FileSystemRouter` 增加：`removePath` / `renamePath` / `copyPath` / `sameMount` / `lastError`，
  根据 mount 派发到 SftpClient 或 RealFileSystem；SMB 暂返回 "not implemented"
- `MainWindow` 三个入口去掉远程拦截：
  - `onRenameEntry`：远程用 `router.exists/isDirectory/renamePath`，覆盖时用 `router.removePath` 先清
  - `onDelete`：远程走 `router.removePath`，进度按"条目"粒度推进（fs.countEntries 不懂 sftp:// 路径）

**注意**：SFTP 协议没有 server-side copy 原语，`copyFile` 是通过 SSH 通道
read 一边 write 另一边（按 32K 块流式）。失败时尝试 unlink 半成品。

## 调研：SFTP 文件复制 / 删除 / 改名 已经接通

用户提出"SFTP 需要支持文件复制、删除、改名"。调研结论：**已实现，无需新增代码**。

- `SftpClient` 早已具备 `removeFile / removeEmptyDirectory / removeRecursive / renamePath / makeDirectory / copyFile / copyRecursive` 全套写操作 API（基于 libssh2_sftp_*）。
- `FileSystemRouter` 已暴露 `removePath / renamePath / copyPath / sameMount / lastError`，并把 SFTP 路径派发到对应 SftpClient。
- `MainWindow::onDelete / onRenameEntry / onPaste` 均已按 `router.isRemote(path)` 分支：
  - 远程：调 `router.removePath / renamePath / copyPath`，进度按"条目"粒度推进
  - 同 mount 内 cut/copy 走 server-side rename / copyRecursive
  - 跨边界（本地↔远程 / 跨 mount）显式弹"Not supported yet"
- 仅 SMB 写操作未实现（router 里返回错并设 lastError("SMB ... is not implemented yet")）。

本次无代码改动，编译通过验证（cmake --build build -j）。

**关键调用链**：
- 删除：`MainWindow::onDelete` → `FileSystemRouter::removePath` → `SftpClient::removeRecursive`
- 改名：`MainWindow::onRenameEntry` → `FileSystemRouter::renamePath` → `SftpClient::renamePath`
- 复制：`MainWindow::onPaste` → `FileSystemRouter::copyPath` → `SftpClient::copyRecursive`

## 完成：跨边界传输（本地↔SFTP 上传 / 下载）

放开 onPaste / onDropCopy 的"本地↔远程"硬拦截，让快捷管理器登入 SFTP 后双面板可双向传输。

**对齐结论**（用户确认）：① SMB 上传下载不做；② Cut 跨边界 = 传完删源；③ 进度走 byte-level。

### 改动
1. **`SftpClient`**：补全 `uploadFile / uploadRecursive / downloadFile / downloadRecursive`（声明早已存在但实现空缺），按 32 KB 块流式 read+write，新增 `TransferProgressFn(currentName, bytesDone, bytesTotal)` byte-level 进度回调。新增静态 `localTotalBytes(path)` 与成员 `remoteTotalBytes(path)` 用作预扫。失败时 `libssh2_sftp_unlink` / `QFile::remove` 半成品。
2. **`FileSystemRouter`**：
   - 新增 `transferAcross(src, dst, progress, total, *done)` —— 自动判方向：local→sftp 走 uploadRecursive，sftp→local 走 downloadRecursive；SMB 任一侧或同侧均返回 false。
   - 新增 `totalBytes(path)` 派发到 `SftpClient::localTotalBytes` / `remoteTotalBytes`。
3. **`MainWindow::onPaste`**：去掉"本地↔远程"硬拦截（保留"跨不同远程 mount"拦截）。新增跨边界分支：预扫 totalBytes → QProgressDialog（>INT_MAX 时按 KB 折算）→ 逐条目调 `router.transferAcross`，progressFn 回调里更新进度并支持取消。**Cut 跨边界传输成功后调 `router.removePath` 删源**，与本地跨设备 move 范式一致。
4. **`MainWindow::onDropCopy`**：同样放开本地↔SFTP 边界，复用相同的 byte-level 进度模式；远程→远程同 mount 拖拽暂保留拒绝。

### 关键设计
- `bytesDone` 是指针，递归调用之间共享同一个计数器 → UI 拿到的是平滑增长曲线，不会因为递归子目录"重置"。
- 取消语义：progressFn 返回 false → SftpClient 立刻 break + setError("canceled by user") + unlink 半成品。
- 进度对话框上限 int 范围：>INT_MAX 时折算成 KB，避免溢出。
- 失败时把"应有字节量"补齐 bytesDone，避免后续条目进度回退。

`cmake --build build -j` 通过。
